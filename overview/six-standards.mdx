title: "The six documentation standards"
description: "Every piece of documentation must meet all six standards before it's ready to publish."
---

Every piece of content must meet all six standards before it's ready to publish. Use these as your checklist every time you write or review.

<CardGroup cols={2}>
  <Card title="User-focused" icon="user">
    Write for what the user wants to do, not what the tool is. Every heading and section answers one question: what will the user accomplish here?
  </Card>
  <Card title="Trustworthy" icon="shield-check">
    Verify every claim against the live product before publishing. Don't write from memory. If you can't confirm it, don't include it.
  </Card>
  <Card title="Easy to understand" icon="brain">
    Reduce the cognitive load for the reader. Use plain language, active voice, and short sentences. If a sentence is hard to write, it's hard to read.
  </Card>
  <Card title="Easy to find" icon="magnifying-glass">
    Structure content around how users think and move through tasks. Users scan, skip, and search. Make it easy to land in the right place.
  </Card>
  <Card title="Usable" icon="check-circle">
    Write so a user can complete the task unaided, with no guesswork. If a user needs to ask for help after reading your documentation, the documentation isn't done.
  </Card>
  <Card title="Style compliant" icon="pen-ruler">
    Use active voice, second person ("you"), present tense, and plain language throughout. No sentence exceeds 25 words.
  </Card>
</CardGroup>

## The test for each standard

Apply this test to your work before every submission. If any standard fails, revise before publishing.

| Standard | The test |
|---|---|
| User-focused | Does every heading describe a task or outcome, not a feature? |
| Trustworthy | Have you performed every step in the live product before publishing? |
| Easy to understand | Can a developer scan this and act immediately? |
| Easy to find | Can a user who arrives from search follow this section without reading what came before it? |
| Usable | Can a user complete the task from this page alone, with no guesswork? |
| Style compliant | Does every sentence pass the 25-word limit, active voice, and second person checks? |

## Tool-focused vs user-focused writing

This distinction is the most important principle and the easiest to get wrong.

| Tool-focused (wrong) | User-focused (right) |
|---|---|
| Alert Configuration Panel | Configure an alert |
| Camera Management | Add and manage cameras |
| User Permissions | Set up user access |
| Metrics Tool | Count records in a file |

Read every heading and section intro you write. If it describes the product, rewrite it to describe what the user will do or achieve.
