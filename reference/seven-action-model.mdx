Content type frameworks describe what to write. The seven action model describes what users need to accomplish. Use it to decide what to build, then use Diataxis to decide how to write it.

The model comes from the talk "Beyond Content Types: A Human-first Model for Technical Documentation."

## The seven actions

<AccordionGroup>
  <Accordion title="1. Appraise: evaluate, assess">
    Users who want to know whether the product is right for them before committing.

    Documentation that satisfies this: product overview, service description, capability comparison.

    **Metric:** conversion rate from documentation to signup.
  </Accordion>
  <Accordion title="2. Understand: comprehend, learn">
    Users who need to grasp the abstractions and concepts behind the product before they can use it effectively.

    Documentation that satisfies this: conceptual overview, glossary, architecture diagram.

    **Metric:** learning path completion.
  </Accordion>
  <Accordion title="3. Explore: discover, browse, try">
    Users who have signed up and are navigating the product for the first time.

    Documentation that satisfies this: interactive quick starts, sandboxes, guided tours.

    **Metric:** user engagement and feature signups.
  </Accordion>
  <Accordion title="4. Practice: exercise, operate">
    Users who are learning to operate most of the product's features.

    Documentation that satisfies this: tutorials, how-to guides, user guides.

    **Metric:** feature adoption rates.
  </Accordion>
  <Accordion title="5. Remember: recall, retrieve, reference">
    Users who know how to do something but need to look up a specific detail.

    Documentation that satisfies this: reference documentation, command references, keyboard shortcuts.

    **Metric:** return visits and time on reference pages.
  </Accordion>
  <Accordion title="6. Develop: extend, build, integrate">
    Users who are building on top of the product.

    Documentation that satisfies this: API documentation, SDK documentation, integration guides.

    **Metric:** number of integrations created.
  </Accordion>
  <Accordion title="7. Troubleshoot: solve, diagnose, stabilize">
    Users who have a problem and need to fix it. The most neglected action in traditional documentation frameworks.

    Documentation that satisfies this: troubleshooting guides, error references, diagnostic flows.

    **Metric:** decrease in time to resolution and support tickets.
  </Accordion>
</AccordionGroup>

## The doc sandwich

Frameworks and content types are the bread. They hold things together. User needs are the filling. Business goals (OKRs, conversion targets) are the sauce. You need all three. Frameworks alone are not enough.

## How to use this model

Before planning a new documentation section, identify which actions the users coming to that section need to take. Then use the right content type from the Diataxis framework to satisfy each action.

| User action | Content type |
|---|---|
| Appraise | Overview, product overview |
| Understand | Concept, explanation |
| Explore | Quick start, interactive tutorial, sandbox |
| Practice | Tutorial, how-to guide |
| Remember | Reference |
| Develop | API reference, SDK docs |
| Troubleshoot | Troubleshooting guide, error reference |
