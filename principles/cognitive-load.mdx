Cognitive Load Theory was developed by John Sweller in the 1980s. It states that working memory has a limited capacity for processing information. Your job as a writer is to work within that limit.

## The three types of cognitive load

| Type | What it is | Your role |
|---|---|---|
| **Intrinsic** | The inherent difficulty of the subject | Manage it by breaking complex topics into smaller pieces |
| **Extraneous** | Unnecessary effort caused by poor presentation | Eliminate it. This is what bad writing creates. |
| **Germane** | The effort to connect new information to existing knowledge | Support it with good analogies and consistent patterns |

## Ten techniques for reducing cognitive load

Each technique targets extraneous load, the type you can eliminate as a writer. Apply as many as the content calls for.

### 1. Chunking
One idea per sentence. One action per step. Lists and structured formats over dense paragraphs.

### 2. Visual aids
Use diagrams and flowcharts alongside text. Let readers grasp a process visually before they read the details.

### 3. Progressive disclosure
Overview first, quick start next, detailed reference last. Don't dump everything on the reader at once.

### 4. White space
Short paragraphs, bullet points, generous spacing. Dense text is cognitive load.

### 5. Consistent structure
Use the same format across every section. When readers know what to expect, they spend less mental energy navigating.

### 6. Leverage prior knowledge
Connect new concepts to things the reader already knows.

```text
The async/await pattern in JavaScript is similar to Python's coroutines.
```

### 7. Clear navigation
Table of contents, search, glossary, descriptive headings. Readers shouldn't have to hunt for where they are.

### 8. Minimize jargon
Use simpler alternatives where they exist. "The system uses a flexible design" is better than "The system utilizes a polymorphic inheritance model."

### 9. Examples and analogies
Concrete beats abstract every time. "Think of a REST API like a restaurant menu" gives the reader a mental hook before the technical explanation.

### 10. Interactive elements
Collapsible sections, interactive code samples, and tooltips let readers go as deep as they need without overwhelming those who don't.

## The caveman technique

Use this technique when a sentence is too complex or abstract to fix directly.

<Steps>
  <Step title="Strip the sentence to its bare meaning">
    Write it in the shortest, most primitive form you can. Grammar doesn't matter here. You're finding the meaning.

    Original: *"The interface provides the ability to instruct the operating system to render items between event dispatches."*

    Stripped: *"Interface tell operating system. Render item between event."*
  </Step>
  <Step title="Identify the core idea">
    What is the sentence actually saying? Remove abstractions.
  </Step>
  <Step title="Rebuild cleanly">
    Rewrite from the stripped version using plain language, active voice, and second person.

    Rebuilt: *"Use the interface to render items between events."*
  </Step>
</Steps>

Apply this test to your work before every submission. If any standard fails, revise before publishing.

<Note>
The stripped version is never the final output. It's a middle step that forces you to find the meaning. Check the rebuilt version against all six documentation standards before publishing.
</Note>
