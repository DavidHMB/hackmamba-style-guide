title: "Information architecture"
description: "How to organize, label, and connect documentation so users can find what they need."
---

Information architecture is how you organize, label, and connect content so users can find things without getting lost. When structure is predictable, users move through content with less friction and less cognitive load.

## Four key components

| Component | What it covers |
|---|---|
| **Organization systems** | How you group and relate content |
| **Labeling systems** | The words users see in navigation and headings |
| **Navigation systems** | Menus, breadcrumbs, sidebar links, and local navigation |
| **Search systems** | The fallback when navigation doesn't do enough work |

## IA rules

- Design for a predictable structure. Users forgive small imperfections, but they don't forgive surprises in hierarchy.
- Every page is a front door. People arrive from search, Slack, links, or bookmarks, so every page needs clear orientation and obvious next steps.
- Avoid junk-drawer categories like "General," "Resources," or "Other." They create friction and force users to translate your system.
- When structure matches user mental models, users spend less time decoding the system and more time finishing the task.
- Map the full end-to-end flow of a task before you start writing. Organize content to match that flow.

## Information fragmentation

**Information fragmentation** is what happens when content is structured around product features rather than user goals. Users have to jump between sections to complete one task. If a user has to visit three pages to finish one task, the content is fragmented. Reorganize around the task.

<Tip>
The naming conventions in [Headings](/writing-rules/headings) enforce the labeling rules described here: imperative verbs for tasks, noun phrases for concepts, consistent patterns within the same navigation level.
</Tip>

## Sidebar naming

| Use | Avoid |
|---|---|
| Dashboard, Projects, Billing, Team, Settings | General, Resources, Miscellaneous, Admin Stuff, Other |

Labels should make sense out of context. Users often land on a page from search or a shared link.

## Tutorial and guide labels

Use labels that tell users what they will learn or do.

| Pattern | Examples |
|---|---|
| Imperative verbs for tasks | "Set up billing", "Connect your CRM", "Start here" |
| Gerunds or nouns for concepts | "Creating API keys", "Connecting apps" |
| Consistent patterns within a set | Don't mix "How to connect" with "Connecting" at the same navigation level |

## UI product and tool labels

Prefer labels that describe the object or action directly.

**Good:** Save, Filters, Import data, API keys, Integrations, Permissions

**Avoid:** More, Info, Click here, or company-specific jargon the audience doesn't already know
