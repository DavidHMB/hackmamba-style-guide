---
title: "Developer mindset"
description: "How developers read documentation and what they expect from it."
---

Developers use documentation to solve problems. They don't read it for pleasure. Understanding how they think changes how you structure and write everything.

## Developers scan, not read

Developers don't read documentation linearly. They scan for the specific piece they need, copy what they can use, and get back to work. Structure every section so it stands on its own. A developer who lands on a page from a search result shouldn't need to read the pages before it.

## Friction is the enemy

Every unclear step, every missing example, and every wall of text is friction that breaks a developer's flow. Developers are trained to identify friction in a process and eliminate it. When they encounter friction in your documentation, they abandon it and find another way: Stack Overflow, GitHub issues, Discord, or trial and error.

## Match the workflow sequence

If a developer needs to install, configure, and then integrate, your documentation follows that exact order. Don't make them jump between pages to complete one task. This is information fragmentation, and it's the most common structural problem in developer documentation.

## Put docs where developers already are

Developers live in GitHub, terminals, and IDEs. Docs that live only in a separate web portal create friction. Where possible, bring the docs to the workflow: README files, inline code comments, and CLI help text all count as documentation.

## Write for the moment of need

A developer hitting an error at 2am doesn't want a conceptual overview. They want the exact command or fix. Structure your docs so the right information appears at the right moment in the workflow.

## Support automation

Provide copy-paste commands, code samples that run without modification, and API references that integrate with developer tooling. If a developer has to adapt your example before they can use it, you've added friction.
