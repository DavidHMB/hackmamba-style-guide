---
title: "Technical reviews"
description: "How to run effective technical reviews and what to expect from reviewers."
---

Technical reviews are the most important step in documentation production. Documentation that hasn't been verified by someone who knows the product is a liability.

## Rules for technical reviews

- Engineers who know the product must do the reviews. Not project managers, not product owners.
- Be specific about what's wrong and explain why.
- Meet the deadline. Late reviews cascade and delay everything.
- Treat technical reviews exactly like a code review.

## What to avoid

- Don't change deadlines without telling the writer immediately
- Don't delay technical reviews
- Don't disregard the writer's advice on structure and clarity
- Don't share documents that aren't relevant to the project

## Assign a doc caretaker

Before the writer leaves a project, assign a doc caretaker. This is someone on the team who owns keeping the documentation updated after launch. Without ownership, documentation decays.

## What the reviewer checks

A technical reviewer focuses on accuracy and completeness, not style. They verify:

- Every step produces the described result when followed in the live product
- Every code sample runs without errors
- Every parameter, field name, and status code is correct
- Known limitations are accurately described
- Nothing is missing that would leave a user without a path forward

The writer handles style compliance. The reviewer handles technical accuracy.

## For writers: preparing for review

Before sending documentation for technical review:

- Complete your own self-review against the full editing checklist
- Highlight any sections where you were uncertain and flagged for verification
- Note which sections are AI-assisted so the reviewer can evaluate them carefully
- Test every step yourself before sending for review

A technical review is not a substitute for self-review. It's a second layer of verification.
